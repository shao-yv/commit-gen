# 智能 Git 提交信息生成器 项目计划书（更新版）

## 1. 项目概述
本项目旨在开发一个基于大语言模型的命令行工具，能够自动读取 Git 仓库的代码变更（diff），并调用 AI 模型生成符合 Conventional Commits 规范的提交信息。最终产出包括：
- 稳定可用的命令行工具
- 软件著作权证书
- 一篇学术论文
- 完整的 GitHub 代码仓库

该成果将用于考研复试，展示软件工程实践与科研潜力。

## 2. 技术栈
| 类别 | 工具/库 | 用途 |
|------|--------|------|
| 编程语言 | Python 3.13 | 核心开发语言 |
| 版本控制 | Git / GitHub | 代码管理与协作 |
| Git 交互 | GitPython | 读取仓库信息、获取 diff |
| AI 服务 | 硅基流动 API | 调用大模型（Qwen/DeepSeek-V3） |
| 嵌入模型 | BAAI/bge-large-zh-v1.5 | RAG 检索中的文本向量化 |
| 向量数据库 | ChromaDB | 存储和检索示例 |
| 环境变量 | python-dotenv | 安全管理 API Key |
| 命令行 | argparse | 解析用户参数 |
| 文档 | Markdown | 项目计划、开发日志、README |

## 3. 阶段划分与时间计划

### 阶段1：环境搭建与 API 接入 ✅（已完成）
- **任务**：安装 Python、VS Code、Git；注册硅基流动账号；获取 API Key；编写测试脚本，成功调用 API。
- **时间**：2 天
- **产出**：可运行的 API 调用示例，`test_api.py`，`.env` 配置文件。

### 阶段2：基础功能开发 ✅（已完成）
- **任务**：
  - 使用 GitPython 读取当前仓库的未暂存变更（`git diff`）。
  - 调用 API 生成提交信息，并输出到控制台。
  - 添加 `argparse` 命令行参数支持（`--path` 指定仓库路径，`--stage` 查看暂存区 diff）。
  - 完善错误处理（无 Git 仓库、无变更、API 超时等）。
  - 增加降级方案：当 API 不可用时，生成通用提交信息。
- **时间**：5 天
- **产出**：稳定可用的命令行工具 `commit_gen.py`，支持基本参数和健壮性。

### 阶段3：代码模块化重构与软著准备 🔄（进行中）
- **任务**：
  - 将单文件 `commit_gen.py` 拆分为多个模块（`git_handler.py`、`llm_client.py`、`prompt_builder.py`、`preprocessor.py`、`cli_parser.py`、`config.py`、`main.py`）。
  - 为克隆检测预留接口（`preprocessor.py` 中定义分析函数）。
  - 增加单元测试，完善注释，使代码行数达到 3000 行以上。
  - 撰写《软件说明书》《用户操作手册》，整理源代码，准备软著申请材料。
- **时间**：1~2 周
- **产出**：模块化代码、软著申请材料。

### 阶段4：实验设计（Few-shot 对比）与数据收集
- **任务**：
  - 设计研究问题（例如“不同提示词策略对提交信息质量的影响”）。
  - 从 CommitBench 数据集中抽取测试样本。
  - 实现 Few-shot 提示词（手动选择示例），对比“无示例”“简单示例”“结构化示例”的效果。
  - 收集评估数据：使用 BLEU、ROUGE 等自动指标，以及人工评分。
- **时间**：2~3 周
- **产出**：实验数据、图表，为论文提供核心内容。

### 阶段5：RAG 原型实现与评估（可选扩展）
- **任务**：
  - 构建小型检索库：从 CommitBench 中抽取 Python 项目的 diff-提交信息对。
  - 使用嵌入模型（如 `BAAI/bge-large-zh-v1.5`）将 diff 转为向量，存入 ChromaDB。
  - 实现检索逻辑：给定新 diff，检索最相似的 k 条示例，拼接到提示词中。
  - 对比 RAG 与 Few-shot、无示例的效果，评估质量提升。
- **时间**：1~2 周
- **产出**：RAG 增强版工具，额外的实验数据。

### 阶段6：论文写作与投稿
- **任务**：
  - 确定论文题目（例如《基于大语言模型的代码提交信息自动生成：提示词策略与 RAG 增强》）。
  - 撰写论文：摘要、引言、相关工作、系统设计、实验评估、结论。
  - 选择目标期刊或会议（如《计算机与现代化》、CCF 中国软件大会学生论文）。
  - 根据反馈修改，完成投稿。
- **时间**：2~3 周
- **产出**：论文初稿，并完成投稿。

### 阶段7：克隆检测扩展（可选，视时间而定）
- **任务**：
  - 在 `preprocessor.py` 中实现轻量级克隆检测（如基于 Token 序列的 Jaccard 相似度）。
  - 将检测结果融入提示词，生成更智能的提交信息（如“统一修复重复代码”）。
- **时间**：1 周
- **产出**：功能增强版工具，论文可增加相应实验。

## 5. 关键风险与应对
| 风险 | 应对策略 |
|------|---------|
| API 服务不稳定或超时 | 增加重试机制，使用响应更快的模型（如 Qwen-7B） |
| 免费额度不足 | 使用永久免费模型，或小额充值 |
| 论文写作不顺利 | 提前阅读相关文献，使用 AI 辅助润色和结构设计 |
| 时间紧张 | 优先保证核心功能（模块化+实验），扩展功能酌情取舍 |
| 技术知识不足 | 制定学习计划，主动理解代码，随时向知南提问 |
